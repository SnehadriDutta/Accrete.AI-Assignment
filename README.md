# Narrative Engine (NE): Topic Deduplication & Semantic Search

## Overview
This repository contains the solution for processing, analyzing, and searching messy social media data generated by a Narrative Engine (NE) model. The project is divided into two primary pipelines:
1. **Data Pipeline (Part 1):** Ingesting JSON data across overlapping 30-day windows and deduplicating identical topics.
2. **Search Pipeline (Part 2):** An intent-based semantic search engine capable of retrieving topics based on a user's complex natural language query, built to scale without relying on expensive $O(N)$ LLM operations.

## Architecture & Methodology

### Part 1: Topic Deduplication
The dataset consists of social media topics generated across three independent runs with overlapping time windows. Consequently, the same real-world events generate highly similar, redundant topics across the files.

**The Solution:**
To consolidate these topics without using an LLM (to simulate cost restrictions), the pipeline implements a heuristic **Jaccard Similarity (Intersection over Union)** approach.
* **Extraction:** For each valid topic, the `title`, `summary`, and `key_entities` are extracted and tokenized.
* **Comparison:** Tokens are compared against a running pool of processed topics. If the Jaccard score exceeds a `0.5` threshold, the topic is flagged as a duplicate and discarded. 
* **Output:** A clean, consolidated list of "Master Topics."

### Part 2: Intent-Based Semantic Search
Users need the ability to search for topics using plain text queries that describe both *what* they want to see and *why* they want to see it (e.g., "I want to see topics about the Super Bowl... I am interested in seeing the political controversy"). Brute-force LLM comparisons are prohibited due to scaling constraints.

**The Solution:**
The system uses an **Asymmetric Semantic Vector Search** architecture. 
* **Data Embedding:** The Master Topics are concatenated into dense strings (Title + Summary + Entities) and embedded using `multi-qa-MiniLM-L6-cos-v1`.
* **Why this model?** Standard semantic models evaluate if two paragraphs are identical. The `multi-qa` model is fine-tuned specifically for asymmetric retrieval—mapping short, conversational user queries to long, dense document summaries, naturally boosting relevance scores.
* **Execution:** Queries are embedded using the same model. A Cosine Similarity calculation instantly retrieves the top-K most semantically aligned topics in sub-linear time, allowing the system to scale to millions of records.

## Validation & Evaluation Strategy

To prove the efficacy of the retrieval system, a two-pronged evaluation framework is proposed:

1. **Quantitative Baseline (IR Metrics):** Using a "Golden Dataset" of manually mapped queries to target `topic_id`s, we track **Recall@K** (to ensure the correct topic appears in the top retrieved results) and **Mean Reciprocal Rank (MRR)** (to evaluate ranking quality).
2. **Qualitative Validation (LLM-as-a-Judge):** Because the user's "why" (intent) is subjective, a secondary LLM script is used to automate semantic validation. The judge evaluates the user's original query against the retrieved topic's summary, scoring it from 1-5 based strictly on *Topical Relevance* and *Intent Satisfaction*.

## Setup & Installation

## Setup & Installation

**Prerequisites:**
* Python 3.9+
* `uv` package manager installed (If not installed: `pip install uv` or via curl/brew. See [astral-sh/uv](https://github.com/astral-sh/uv) for details)

**1. Create and Activate a Virtual Environment:**
`uv` can create virtual environments almost instantly. 
```bash
uv venv

# On macOS and Linux:
source .venv/bin/activate

# On Windows:
.venv\Scripts\activate
```
**2. Run application:**
```
uv run [filename.py]
```
